# Scalable Spring Boot + Firebase Backend Architecture

This document outlines the architecture, design, and implementation strategy for a scalable backend system using Spring Boot and Firebase (Firestore as the primary database), completely bypassing Firebase Authentication in favor of a custom JWT-based authentication system.

## 1. Clean Architecture & Best Practices

To ensure scalability and maintainability, the application will follow a strict layered **Clean Architecture**:

*   **Controller Layer (API/Presentation):** Handles incoming HTTP requests, validates request payloads, and routes them to the appropriate service. Returns localized and formatted responses (e.g., standard API response wrapper).
*   **Service Layer (Business Logic):** Contains all core business logic. It handles password hashing, JWT generation, OTP generation, and business validations. It acts as an intermediary between controllers and repositories.
*   **Repository Layer (Data Access):** Abstraction over Firebase. This layer handles all direct interactions with Firestore, executing CRUD operations and queries using the Firebase Admin SDK.
*   **Configuration Layer:** Holds configurations for Firebase initialization, Security (Spring Security), JWT filters, and Bean definitions.
*   **Exception Handling:** A global exception handler (`@ControllerAdvice`) to gracefully catch exceptions and return standardized error JSONs.

### Best Practices for Spring Boot + Firebase
1.  **Firebase Admin SDK:** Interact with Firebase exclusively via the Firebase Admin SDK. Do not use the client SDKs on the backend.
2.  **Singleton Initialization:** Initialize the Firebase App instance once during application startup.
3.  **Async Operations:** Firestore provides `ApiFuture`. Wrap them in `CompletableFuture` or reactive constructs if the traffic scales highly, though standard blocking waits are acceptable for MVP.
4.  **Pagination:** Use Firestore cursors (`startAfter`) rather than fetching entire collections.

## 3. Firebase Setup & Credentials

### Required Setup
1.  Go to the [Firebase Console](https://console.firebase.google.com/).
2.  Create a project & navigate to **Project Settings > Service Accounts**.
3.  Click **Generate new private key**. This downloads a JSON file containing your service account credentials.

### Configuration
Never hardcode credentials or commit the JSON file to version control.

**`application.yml`**
```yaml
firebase:
  credentials:
    path: classpath:firebase-service-account.json # OR load via env var for prod
  database:
    url: https://<PROJECT_ID>.firebaseio.com # Needed if using Realtime DB, optional for Firestore
  projectId: your-project-id
  
jwt:
  secret: YOUR_VERY_LONG_AND_SECURE_SECRET_KEY
  expirationMs: 86400000 # 24 Hours
```

## 4. Scalable Data Model (User Schema)

To accommodate frequent schema evolution and maintain NoSQL flexibility, we treat the `User` object with a flexible attributes map. 

**Collection:** `users`
**Document ID:** Auto-generated by Firestore OR custom predictable ID (e.g., standardizing on UUID or email-hash if collisions are managed, but Firestore Auto-ID is safest).

**Schema (`User.java` concept):**
```json
{
  "id": "abc123xyz890",
  "email": "user@example.com",
  "passwordHash": "$2a$10$EixZaYVK1fsbw1ZfbX3OXePaWxn96p36WQoeG6Lruj3vjQ...",
  "status": "ACTIVE", // ACTIVE, INACTIVE, LOCKED
  "auth": {
    "resetToken": "otp_or_uuid_string",
    "resetTokenExpiry": 1698765432000
  },
  "metadata": {
    "createdAt": 1698760000000,
    "updatedAt": 1698760000000,
    "lastLoginAt": 1698760000000,
    "schemaVersion": 1 // Critical for migrations
  },
  "profile": {
    "firstName": "John",
    "lastName": "Doe",
    "phone": "+1234567890"
  },
  // Flexible block for future attributes without breaking existing queries
  "attributes": {
    "preferences": { "theme": "dark" },
    "loyaltyPoints": 150
  }
}
```

## 5. Authentication Flow (Custom JWT)

We will use standard **BCrypt** for password encryption (salt inherently included) and **JWT (JSON Web Tokens)** for stateless authentication.

### Token Verification Flow Diagram
```text
Client                  Spring Boot (Auth Filter)         Controller
  │                               │                            │
  │── 1. POST /api/auth/login ───▶│                            │
  │     (email, password)         │                            │
  │                               │── 2. Verify BCrypt Hash    │
  │                               │── 3. Generate JWT          │
  │◀── 4. Return JWT ─────────────│                            │
  │                               │                            │
  │── 5. GET /api/users/me ──────▶│                            │
  │   (Header: Auth: Bearer <JWT>)│── 6. Validate Signature    │
  │                               │── 7. Extract UserID        │

  │                                                            │── 9. Execute Logic
  │◀── 10. Return 200 OK ──────────────────────────────────────│
```

## 6. Forgot Password Flow

1.  **Request Reset:** User submits their `email`.
2.  **Generate OTP/Token:** The backend generates a secure 6-digit OTP (or UUID token) and an expiration timestamp (e.g., 15 mins).
3.  **Store in Firestore:** Update the user's document under the `auth` nested object with `resetToken` and `resetTokenExpiry`.
4.  **Send Email:** Backend integrates with an email provider (e.g., AWS SES, SendGrid, Mailgun) to email the OTP to the user.
5.  **Verify & Reset:** User submits `email`, `OTP`, and `newPassword`.
6.  **Validation:** Backend checks if `email` exists, if `resetToken` matches the OTP, and if `resetTokenExpiry` > current time.
7.  **Update Hash:** If valid, generate a new BCrypt hash for `newPassword`, update the DB, and clear the `auth.resetToken` fields.

## 7. API Endpoints

| Method | Endpoint | Description | Request Body | Response |
| :--- | :--- | :--- | :--- | :--- |
| `POST` | `/api/v1/auth/signup` | Register new user | `{ email, password, firstName, lastName }` | `{ userId, message }` |
| `POST` | `/api/v1/auth/login` | Authenticate user | `{ email, password }` | `{ accessToken, expiresIn }` |
| `POST` | `/api/v1/auth/forgot-password`| Initiate pass reset | `{ email }` | `{ message: "OTP sent" }` |
| `POST` | `/api/v1/auth/reset-password` | Finalize pass reset | `{ email, otp, newPassword }` | `{ message: "Password updated" }` |
| `GET`  | `/api/v1/users/me` | Get current user | *JWT in Header* | `{ profile, email, ... }` |

> [!IMPORTANT]
> The signup endpoint must check for existing emails to prevent duplicates. Since Firestore doesn't enforce strict unique constraints like SQL natively across fields without specific architectural patterns, you must query for the email before inserting, or use the email as the Firestore Document ID (e.g., `users/user@example.com`).

## 8. Strategy for Schema Evolution & Frequent Changes

Firestore is inherently schema-less, but the backend application expects structure. To handle frequent evolution safely:

1.  **`schemaVersion` field:** Always include a `schemaVersion` integer in your documents. 
    *   *v1* has `name`.
    *   *v2* splits it into `firstName` and `lastName`.
2.  **Lazy Migration (Read-Time):** Write logic in your repository mapper where, upon reading a `v1` document, it maps `name` to `firstName` on the fly, updates the document to `v2` in the background, and returns the converted object.
3.  **Batch Scripts (Write-Time):** For sweeping changes, write a standalone Node.js or Spring Boot background job that fetches all documents of `< targetVersion` in batches of 500, transforms them, and commits the batch writes back to Firestore.
4.  **Flexible Attributes Map:** Store volatile, frequently changing fields (like user preferences) inside an unstructured `attributes` map (Firestore `Map` type) rather than as strict properties on the Java entity. Map this to a `Map<String, Object>` in Spring.

## 9. Security, Monitoring & Scaling

### Security Best Practices
*   **Rate Limiting:** Implement Bucket4j or Redis-based rate limiting on `/api/v1/auth/*` endpoints to prevent brute-force attacks on login and OTP generation.
*   **Password Hashing:** Use `BCryptPasswordEncoder` mapped to Spring Security. Set strength to at least `10`.
*   **Input Validation:** Use `spring-boot-starter-validation` (`@Valid`, `@NotBlank`, `@Email`) on all Request DTOs.
*   **CORS:** Explicitly whitelist frontend domains in `SecurityConfig`. Do not use `*` in production.

### Logging & Error Handling
*   **Structured Logging:** Use `Logback` outputting in JSON format. Ensure PII (passwords, JWTs, plain emails) is masked or excluded from logs.
*   **Global Exception Handling:** Catch all standard exceptions. Return a unified response structure: 
    `{ "timestamp": "...", "status": 400, "error": "Invalid input", "message": "Email is required" }`

### Future Scalability (Microservices)
*   As the application grows, the monolithic Spring Boot app can be containerized using Docker and deployed to a managed Kubernetes service (like GKE) or Serverless (like Google Cloud Run).
*   If features expand massively, the Auth components can be split into an `Auth Service`, and other business logic into domain-specific microservices. JWTs facilitate this well because any internal service with the public key/secret can validate the token immutably without hitting the `Auth Service` again.
